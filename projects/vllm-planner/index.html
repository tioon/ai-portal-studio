🧠 vLLM 메모리 사용량 인터랙티브 가이드

GPU 위에서 vLLM으로 LLM을 서빙할 때, gpu_memory_utilization·max_model_len·tensor_parallel_size·양자화 등 설정값에 따라 모델 가중치 / KV 캐시 / 실행 오버헤드 / 호스트 RAM이 어떻게 배분되는지 직접 확인할 수 있고, 이제는 실제 용도와 동시 사용자 수를 넣으면 추천값도 자동으로 채워집니다.

자동 설정 플래너
실제 용도, 동시 사용자 수, 원하는 응답속도를 넣으면 배포용 핵심 vLLM 값을 바로 다시 잡습니다. 이 값은 아래 메모리 시뮬레이터와 배포 명령어에 함께 반영됩니다.

1. GPU 메모리는 어디에 쓰이는가 개요

vLLM 프로세스가 GPU 한 장에서 점유하는 메모리는 크게 4가지로 나뉩니다.

모델 가중치

실제 체크포인트 파일 총합. Tensor Parallelism(TP)으로 GPU 수만큼 분할되고, 양자화(AWQ/GPTQ/FP8/INT4)로 바뀌는 경우엔 해당 체크포인트 자체의 크기를 기준으로 다시 계산해야 합니다.

KV 캐시

PagedAttention이 블록 단위로 관리하는 어텐션 캐시. gpu_memory_utilization이 정한 여유 메모리의 대부분을 차지하며, 동시 처리 가능한 요청 수를 결정합니다.

CUDA Graph / 활성화 버퍼

추론 속도를 위해 캡처하는 CUDA Graph, 그리고 forward pass 중간 활성화 값. enforce_eager=True로 끄면 줄어들지만 속도가 느려집니다.

프레임워크 오버헤드

PyTorch/CUDA 컨텍스트, NCCL 통신 버퍼(TP 사용 시) 등 vLLM이 직접 제어하지 않는 고정 오버헤드.

2. 메모리 시뮬레이터 인터랙티브

아래 값을 조절하면 오른쪽 그래프와 통계가 실시간으로 갱신됩니다. 위 플래너에서 고른 용도/동시 사용자/응답속도에 따라 KV 캐시 도넛도 함께 바뀝니다. 가중치는 프리셋 기본값을 불러오지만 언제든 직접 수정할 수 있습니다.

5. 이 계산기가 다루지 않는 요소 한계

위 시뮬레이터는 VRAM 사용량을 좌우하는 가장 비중이 큰 설정들을 다루지만, 실제 vLLM 배포에는 아래처럼 추가로 메모리에 영향을 주는 요소들이 있습니다 (수치는 반영되지 않음).

LoRA 어댑터 (enable_lora, max_loras, max_lora_rank)

동시에 로드하는 LoRA 어댑터 수 × rank에 비례해 별도 GPU 메모리 풀이 추가로 필요합니다.

스펙큘레이티브 디코딩 (speculative_config)

드래프트 모델의 가중치와 자체 KV 캐시가 추가로 GPU에 상주해야 합니다.

멀티모달 인코더 (비전/오디오 타워)

이미지·비디오·오디오 인코더 가중치와 처리 중 생성되는 피처 캐시가 별도로 메모리를 차지합니다.

어휘(vocab) 크기 · 임베딩

vocab_size가 매우 큰 모델(예: 15만 토큰 이상)은 embedding/lm_head 테이블 자체가 수백 MB~수 GB를 차지할 수 있습니다.

CUDA Graph 캡처 배치 크기 목록

compilation_config에서 캡처하는 배치 크기 종류가 많을수록 CUDA Graph 메모리가 이 계산기의 근사치보다 커질 수 있습니다.

양자화 그룹 크기 · 캘리브레이션 메타데이터

AWQ/GPTQ의 group_size, scale/zero-point 저장 방식에 따라 실제 가중치 메모리가 근사치와 조금씩 달라집니다.

NCCL/커스텀 all-reduce 버퍼

TP·PP 규모, 통신 백엔드(NCCL vs custom all-reduce)에 따라 통신 버퍼 크기가 달라집니다.

MLA/하이브리드 어텐션 압축률

DeepSeek·Kimi 계열의 MLA, Qwen3.6의 선형 어텐션처럼 압축된 캐시 구조는 표준 GQA 공식보다 실제 캐시가 더 작을 수 있어 이 계산기 수치는 상한 근사치입니다.

6. 실전 팁

OOM이 날 때

gpu_memory_utilization을 낮추거나 max_model_len/max_num_seqs를 줄이세요. TP를 늘려 GPU 여러 장에 가중치를 분산하는 것도 방법입니다.

동시 요청이 많을 때 (Preemption 발생)

gpu_memory_utilization을 올리거나 tensor_parallel_size를 늘려 KV 캐시 풀을 키우세요.

속도가 중요할 때

enforce_eager는 끄고(기본값) CUDA Graph를 활용하세요. 메모리 여유가 있다면 FP8 KV 캐시로 캐시 용량을 2배 늘릴 수 있습니다.

긴 컨텍스트가 필요할 때

max_model_len을 늘리는 대신 GQA(적은 KV head) 모델을 쓰거나 FP8 KV 캐시, 큰 block_size로 오버헤드를 줄이세요.

GPU가 여러 장인데 NVLink가 없을 때

PCIe/이더넷 환경에서는 tensor_parallel_size를 1로 두고 pipeline_parallel_size로만 분산하세요. PP는 스테이지 경계에서만 통신하므로 대역폭이 낮아도 상대적으로 안전합니다.

여러 노드(멀티 서버)로 확장할 때

TP는 반드시 노드 내 NVLink 도메인 안에서만 사용하고, 노드 간 확장은 PP로 하세요. 예: 노드당 8장 NVLink + 노드 4개 = TP8 × PP4.

양자화를 고를 때

AWQ/GPTQ INT4는 메모리는 크게 아끼지만 커널 지원·정확도 손실을 확인하세요. 정확도가 중요하면 FP8(가중치+KV 캐시)부터 시도하는 것이 안전한 절충안입니다.

모델이 GPU 1장에 안 들어갈 때

cpu_offload_gb로 일부 가중치를 CPU RAM에 두거나, TP/PP를 늘려 여러 GPU에 나누세요. cpu_offload는 속도 저하가 크므로 임시방편에 가깝습니다.

처리량(throughput) vs 지연시간(latency) 트레이드오프

max_num_seqs·max_num_batched_tokens를 늘리면 처리량은 오르지만 개별 요청의 지연시간(특히 TTFT)은 늘어날 수 있습니다. 서비스 SLA에 맞게 조정하세요.

MoE 모델을 배포할 때

총 파라미터(전체 전문가)가 VRAM 용량을 결정하므로, 활성 파라미터만 보고 GPU를 산정하면 안 됩니다. 총 파라미터 기준으로 TP/PP 규모를 계산하세요.

메모리를 정확히 알고 싶을 때

이 계산기는 근사치입니다. 실제 배포 전에는 --gpu-memory-utilization을 보수적으로(예: 0.85) 설정해 첫 실행 후 nvidia-smi로 실측하고 점진적으로 올리세요.

짧은 프롬프트·대량 동시 요청 서비스

챗봇처럼 max_model_len이 짧고 요청 수가 많은 경우, block_size를 작게(8) 유지하면 내부 단편화를 줄여 더 많은 시퀀스를 동시에 캐싱할 수 있습니다.

긴 문서 요약 등 소수·고정 길이 요청 서비스

동시 요청 수는 적고 max_model_len이 매우 긴 경우, block_size를 크게(64~128) 하면 블록 테이블 오버헤드가 줄어 효율이 좋아집니다.

가중치 메모리는 사용자가 입력한 체크포인트 파일 크기를 기준으로 계산하고, 프리셋은 초기값만 제공합니다. KV 캐시·오버헤드·호스트 RAM은 근사 추정입니다. 실제 값은 vLLM 버전, CUDA 커널, 모델 아키텍처에 따라 달라질 수 있으니 배포 전 nvidia-smi 및 vLLM 로그로 검증하세요.
